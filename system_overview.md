# Interview Preparation Platform

## System Overview (Architecture-Level Specification)

---

## 1. System Purpose

This system is an institutional interview preparation platform.

It processes structured application PDFs and generates non-evaluative interviewer preparation reports.

The system performs:

* Deterministic structural extraction
* Canonical structured representation building
* Single-pass LLM synthesis over structured data
* Policy-constrained output validation

The system does **not** perform applicant evaluation.

---

## 2. Architectural Identity

This system is defined by the following architectural invariants:

1. Deterministic-first processing
2. Single LLM synthesis boundary
3. Canonical representation as the sole source of truth
4. Strict non-evaluative output
5. Traceability of all generated content
6. Infrastructure-agnostic logical core
7. Versioned internal representation

These invariants must remain true across all deployment stages.

Any change that violates these invariants constitutes a system redesign.

---

## 3. Deterministic-First Rule

All structural extraction must occur without LLM involvement.

Deterministic extraction includes:

* Layout parsing
* Section detection
* Field extraction
* Academic record parsing
* Test parsing
* Essay extraction
* Activity extraction
* Cross-reference detection
* Timeline construction
* Integrity analysis

The LLM must never:

* Parse raw PDF directly
* Infer document structure
* Perform schema detection
* Normalize grades
* Compute trends
* Perform evaluation logic

The LLM receives only the canonical structured representation.

---

## 4. Single LLM Boundary

The system makes exactly one LLM synthesis call per processed application.

The system must not:

* Use multiple chained LLM calls
* Use recursive reasoning loops
* Use self-reflection cycles
* Use autonomous tool-calling
* Use LangChain, LangGraph, or similar orchestration frameworks
* Introduce dynamic reasoning graphs

All synthesis must occur in a single controlled invocation.

---

## 5. Non-Evaluative Constraint

The system must not:

* Score applicants
* Rank applicants
* Compare applicants
* Label strengths or weaknesses
* Flag areas of concern
* Infer academic quality
* Predict admissions outcomes
* Normalize grades across systems
* Introduce evaluative adjectives
* Apply comparative language

The system may only:

* Surface structured information
* Highlight explicit cross-references
* Provide neutral discussion anchors
* Generate traceable interviewer prompts

---

## 6. Canonical Representation as Source of Truth

All extracted information must be consolidated into a versioned canonical representation.

The canonical representation must:

* Be collection-based (no rigid fixed academic-level keys)
* Be extensible
* Be backward-compatible across versions
* Separate identifiers (PII) from extracted content
* Separate extracted content from synthesis output
* Include canonical_version metadata

The canonical representation is the only data input to LLM synthesis.

Raw PDF content must not be passed to the LLM.

---

## 7. Traceability Requirement

Every element generated by the LLM must:

* Be directly traceable to content within the canonical representation
* Not introduce new facts
* Not infer implicit traits
* Not add external knowledge
* Not generalize beyond explicit content

If a statement cannot be mapped to canonical data, it must not be generated.

---

## 8. Logical Pipeline (Stage-Agnostic)

The logical processing flow is:

1. PDF ingestion
2. Deterministic layout extraction
3. Section boundary detection
4. Domain-specific structured extraction
5. Cross-reference detection
6. Timeline construction
7. Integrity analysis
8. Canonical representation assembly
9. Single LLM synthesis call
10. Policy validation of generated output
11. Final report generation

This pipeline remains logically constant across all deployment stages.

Infrastructure may evolve; this pipeline must not.

---

## 9. Logical Components

The system contains the following logical components:

* Orchestrator
* Deterministic Extraction Agents
* Canonical Representation Builder
* LLM Synthesis Agent
* Policy Guard

These are logical modules.
They are not necessarily equivalent to network-level microservices.

---

## 10. Infrastructure Independence

The logical system defined in this document is independent of:

* Database choice
* File storage method
* Queue mechanism
* Containerization
* Reverse proxy usage
* Cloud deployment topology

Infrastructure may evolve through stages, but the logical architecture must remain unchanged.

---

## 11. Out of Scope

The system must not implement:

* Admissions decision logic
* Scoring algorithms
* Predictive modeling
* Cross-applicant comparison
* Adaptive model learning
* Reinforcement loops
* Historical outcome learning
* Behavioral inference

---

## 12. Evolution Principle

The system evolves through infrastructure stages:

* Local core engine
* Structured MVP
* Async processing
* Storage hardening
* Service separation
* Security hardening
* Cloud-ready deployment

Across all stages:

* Logical agents remain constant
* Canonical model philosophy remains constant
* LLM contract remains constant
* Non-evaluative constraints remain constant

Infrastructure evolves without altering system identity.

---

## 13. System Identity Statement

This system is:

* Deterministic-first
* Single-LLM-boundary
* Non-evaluative
* Collection-based in representation
* Traceable
* Versioned
* Stage-evolvable

Any implementation that violates these characteristics is non-compliant with this architecture.
