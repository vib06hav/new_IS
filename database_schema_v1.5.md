# `database_schema_v1.5.md`

**(Stage 1.5 — ROS v1 Integrated, No Structural Changes)**

---

## 1. Purpose

This document defines the database schema for Stage 1.

The schema is:

* Minimal
* Container-friendly
* JSONB-driven
* Synchronous-safe
* Deterministic-first compatible

Stage 1.5 introduces ROS v1 as the presentation-layer artifact.
No table structure changes are introduced.

---

## 2. Tables Overview

The database contains four primary tables:

1. `users`
2. `applications`
3. `canonical_records`
4. `synthesis_records`

No new tables are added in Stage 1.5.

---

## 3. users

Unchanged from Stage 1.

Stores:

* User identity metadata
* Authentication reference fields (if applicable)

No changes introduced in Stage 1.5.

---

## 4. applications

Unchanged from Stage 1.

Stores:

* Application ID
* User linkage
* Status
* Confidence score
* Created timestamp

No changes introduced in Stage 1.5.

---

## 5. canonical_records

Unchanged from Stage 1.

### Key Fields

* `id`
* `application_id` (FK)
* `canonical_data` (JSONB)
* `canonical_version`
* `created_at`

### canonical_data (JSONB)

Stores the canonical representation generated by Agents 1–11.

Canonical structure remains:

* Collection-based
* Versioned
* Non-evaluative
* Presentation-agnostic

Stage 1.5 does not modify canonical schema.

No new columns added.

No migration required.

---

## 6. synthesis_records

Unchanged at table level.

### Key Fields

* `id`
* `application_id` (FK)
* `synthesis_output` (JSONB)
* `created_at`

No new columns added.

No new constraints added.

No schema migration required.

---

## 7. synthesis_output (JSONB)

### Stage 1 (Original)

Previously stored:

```json
{
  "snapshot": "string",
  "discussion_focus_areas": "string or array",
  "suggested_questions": "string or array",
  "canonical_version_ref": "string"
}
```

---

### Stage 1.5 (ROS v1)

`synthesis_output` now stores the full ROS v1 document.

Structure:

```json
{
  "report_metadata": { ... },
  "page_1_background_profile": { ... },
  "page_2_academic_and_engagement": { ... },
  "page_3_essays": { ... },
  "page_4_focus_themes": { ... },
  "page_5_question_groups": { ... }
}
```

Full contract defined in:

> ROS v1 

Important:

* JSONB type remains unchanged.
* Column name remains unchanged.
* Table name remains unchanged.
* No migration required.

Only the internal JSON structure evolves.

---

## 8. Schema Stability Rules

Stage 1.5 explicitly does not introduce:

* Additional tables
* Async job tracking tables
* Theme-specific storage tables
* Question-specific storage tables
* Entity mapping tables
* Highlight span tables

All data required for ROS v1 is embedded within:

* `canonical_data`
* `synthesis_output`

The relational schema remains minimal by design.

---

## 9. Forward Compatibility

Because `canonical_data` and `synthesis_output` are JSONB:

* Future ROS versions can evolve without schema migration.
* Canonical versions can evolve independently.
* No rigid SQL constraints are introduced.

Database layer remains stable across Stage 1 → 1.5 transition.
